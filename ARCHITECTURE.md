# Архитектура проекта: Приложение для повышения грамотности

## Концепция
MVP представляет собой псевдо-чатбота (без ИИ), который проводит интерактивные квизы для старших школьников по трем категориям:
1. Орфография
2. Пунктуация
3. Орфография и пунктуация

## Ответы на ключевые вопросы

### 1. Фреймворки и библиотеки
**Next.js** идеально подходит под ваши требования. Использование Next.js App Router (с Server Components и Server Actions) позволит:
- Моментально фетчить обновленные данные из БД при каждом запросе (отсутствие stale data), обеспечивая SSR.
- Держать логику перемешивания (round-robin или псевдорандом) на сервере, не перегружая клиентское устройство.

**Рекомендуемая модульная структура:**
- Разделение на UI-компоненты (кнопки, бабблы сообщений), фиче-компоненты (чат-контейнер) и серверные actions (запросы к БД).

### 2. Взаимодействие с БД и выбор решения
Для MVP разворачивать **PayloadCMS может быть избыточно**, так как это полноценная CMS, требующая поддержки своей инфраструктуры и отдельного деплоя.

**Решения для Self-Hosted развертывания (Dokploy):**
Так как вы планируете деплой на Dokploy, облачные базы теряют свой смысл. Dokploy "из коробки" поддерживает создание баз данных, поэтому лучшим решением будет держать БД рядом с приложением на одном сервере.

1. **PostgreSQL (Native via Dokploy):** Самый чистый и надежный вариант. Вы разворачиваете стандартный контейнер Postgres прямо в интерфейсе Dokploy. Приложение Next.js и БД будут общаться внутри одной локальной сети Docker, что даст нулевую сетевую задержку.
2. **MongoDB (Native via Dokploy):** Если вам ближе NoSQL. Документоориентированная структура отлично подходит для хранения вопросов квиза в формате JSON. Dokploy также поддерживает запуск Mongo в один клик.
3. **SQLite / Turso:** Если хочется максимально легкую БД. SQLite-файл можно хранить прямо в Docker-контейнере Next.js (обязательно примонтировав Volume в Dokploy, чтобы данные не стерлись при редеплое).
4. **Sanity (Облачная Headless CMS):** Если вам всё же нужна визуальная админка для контент-менеджеров, но вы не хотите утяжелять свой сервер развертыванием PayloadCMS. Sanity возьмет на себя хранение контента, а ваш Next.js на Dokploy будет просто ходить к ним по API.

**ORM (Прослойка между БД и кодом)**: Если выберете реляционную БД (Turso, Neon), используйте **Drizzle ORM** (максимально быстрая и легковесная) или **Prisma**.

### 3. Использование Bun вместо npm
**Да, абсолютно.** Bun отлично работает с Next.js.
Преимущества Bun в данном контексте:
- Установка зависимостей в несколько раз быстрее.
- Быстрый запуск скриптов разработки.
- Нативная поддержка TypeScript без дополнительных настроек.

### 4. Валидация данных и уникальный UX
Для обеспечения уникального UX в формате квиза-чатбота:

**Валидация:**
Используйте библиотеку **Zod**. С её помощью можно описать схему данных (например, `QuestionSchema`), чтобы гарантировать, что из БД всегда приходит корректная структура (массив вариантов ответа, индекс правильного ответа, категория), и типизировать API.

**Интерактивность и UX:**
- **Эффект набора текста (Typing Indicator):** Перед появлением вопроса показывайте индикатор (анимация трех точек), как будто собеседник печатает. Это создает эффект живого общения.
- **Мгновенный фидбэк:** При клике на вариант ответа, кнопка сразу красится в зеленый/красный. Если ответ неверный — легкая анимация shake (тряска), как при вводе неверного пароля. Для этого отлично подойдет **Framer Motion**.
- **Объяснения:** После ответа (особенно неверного) бот должен присылать следующее сообщение с правилом русского языка.
- **Геймификация:** Стрики (серии правильных ответов), прогресс-бар вверху экрана, заморозка стрика (как в Duolingo).

## Алгоритм выдачи вопросов (Псевдорандом)
Для того чтобы вопросы не повторялись, можно реализовать следующий подход на стороне сервера (Next.js):
1. У каждого вопроса в БД есть `id`.
2. При начале сессии (`/start`) на клиенте (в `localStorage` или стейте) инициализируется массив пройденных `id`. При каждом ответе этот массив пополняется и отправляется на сервер при следующем запросе.
3. Сервер (Next.js) делает запрос к БД, исключая переданные `id` (используя `NOT IN`), и выбирает случайный вопрос из оставшихся. Это обеспечит неповторяемость до полного прохождения пула вопросов.
