# Пунктуационный конструктор

## Что это за тип задания

`punctuation_constructor` — отдельный тип интерактивного упражнения, где ученик расставляет знаки препинания в предложении.

Задание хранится в общей таблице `exercises`, как и остальные упражнения. Отдельная таблица не используется.

Ключевое поле:

```txt
type = punctuation_constructor
```

## Формат хранения в БД

В `payload` лежит то, что видит ученик:

- токены предложения;
- банк доступных знаков.

Пример:

```json
{
  "tokens": ["Мне", "сказали", "Ждите", "приедет", "другой", "замерщик"],
  "markBank": ["comma", "colon", "quote_open", "quote_close", "period"]
}
```

В `answer` лежит правильная расстановка знаков:

```json
{
  "placements": [
    { "slotIndex": 2, "mark": "colon" },
    { "slotIndex": 2, "mark": "quote_open" },
    { "slotIndex": 3, "mark": "comma" },
    { "slotIndex": 6, "mark": "quote_close" },
    { "slotIndex": 6, "mark": "period" }
  ]
}
```

## Что такое `slotIndex`

`slotIndex` — позиция для знака относительно токенов:

- `0` — перед первым словом;
- `1` — после первого слова;
- `2` — после второго слова;
- `6` — после шестого слова, то есть в конце предложения.

Например, для токенов:

```txt
Мне | сказали | Ждите | приедет | другой | замерщик
```

правильная пунктуация:

```txt
Мне сказали: «Ждите, приедет другой замерщик».
```

Для этого:

- после `сказали` ставятся `:` и `«`;
- после `Ждите` ставится `,`;
- после `замерщик` ставятся `»` и `.`;
- порядок знаков внутри одного слота важен.

## Доступные знаки

Сейчас знаки задаются не символами, а стабильными идентификаторами:

```txt
comma        ,
colon        :
semicolon    ;
dash         —
quote_open   «
quote_close  »
paren_open   (
paren_close  )
period       .
exclamation  !
question     ?
ellipsis     ...
```

Это сделано, чтобы не зависеть от проблем кодировки с русским текстом и типографскими символами.

## Как работает UI

В карточке задания:

- слова отображаются отдельными блоками;
- между словами есть слоты для знаков;
- снизу расположен банк знаков;
- ученик выбирает слот и кликает знак;
- также можно перетащить знак в слот;
- в один слот можно положить несколько знаков подряд, например `:«`;
- поставленный знак можно удалить кликом по нему;
- кнопка `Сбросить` очищает все знаки.

Визуально модель такая:

```txt
[slot 0] Мне [slot 1] сказали [slot 2] Ждите [slot 3] приедет [slot 4] другой [slot 5] замерщик [slot 6]
```

Для прямой речи в примере выше `slot 2` должен содержать два знака:

```txt
:«
```

## Как проверяется ответ

При отправке карточка передаёт:

```json
{
  "type": "punctuation_constructor",
  "placements": [
    { "slotIndex": 2, "mark": "colon" },
    { "slotIndex": 2, "mark": "quote_open" },
    { "slotIndex": 3, "mark": "comma" },
    { "slotIndex": 6, "mark": "quote_close" },
    { "slotIndex": 6, "mark": "period" }
  ]
}
```

Checker сравнивает отправленные `placements` с `answer.placements`.

Ответ считается правильным, если:

- совпадают все слоты;
- совпадают все знаки;
- совпадает порядок знаков внутри каждого слота.

Если ответ неверный, checker возвращает ошибки:

- пропущенный знак;
- лишний знак.

## Формат в админке

Пока админка использует ручной текстовый ввод.

Токены:

```txt
Мне | сказали | Ждите | приедет | другой | замерщик
```

Банк знаков:

```txt
comma, colon, quote_open, quote_close, period
```

Правильные слоты:

```txt
2:colon, 2:quote_open, 3:comma, 6:quote_close, 6:period
```

## Текущие ограничения

Это MVP, а не финальная версия конструктора.

Ограничения:

- ученик не может менять регистр слов;
- поэтому `Ждите` уже должно быть записано с заглавной буквы в `tokens`;
- ученик не может редактировать сами слова;
- нет визуального редактора задания в админке;
- нет объяснения по каждому конкретному слоту;
- нет поддержки альтернативных правильных вариантов;
- проверка строгая: другой порядок знаков считается ошибкой;
- прямую речь можно задать, но UX для составных знаков ещё требует доработки.

## Вопросы для дальнейшего обсуждения

1. Должен ли конструктор работать только со знаками или ещё менять регистр первой буквы прямой речи?
2. Нужно ли разрешить редактирование слов, например исправление орфографических ошибок?
3. Нужны ли альтернативные ответы?
4. Как лучше показывать составные слоты: одной ячейкой `:«` или двумя вложенными мини-блоками?
5. Нужно ли подсвечивать после проверки каждый слот отдельно?
6. Должен ли банк знаков содержать лишние знаки-ловушки?
7. Нужен ли визуальный редактор в админке вместо ручного `slot:mark`?
